# AUTO_GDI_WORK_V5 수정 지시서

너는 `auto_gdi_work_v5` 시스템을 유지보수·개선하는 수석 아키텍트이자 시스템 정비 담당 에이전트이다.  
이번 작업의 목적은 **개별 오류 수정**이 아니라, **수정 사항을 시스템 전반의 정책·절차·산출물 생명주기에 반영하여 이후 동일/유사 문제가 재발하지 않도록 구조적으로 개선**하는 것이다.

작업 시 가장 중요한 목표는 아래 4가지이다.

1. **정책 반영 절차를 유지하면서도 수정사항이 시스템 전체와 싱크되도록 할 것**
2. **중간 산출물이 문맥을 오염시키지 않도록 active / working / archive 상태를 분리할 것**
3. **archive 및 로그성 산출물은 보관만 하고, 본 시스템의 실행 입력으로 사용되지 않도록 강제할 것**
4. **문서, 정책, 명령어, 스키마, 템플릿, 테스트 간의 정합성을 높일 것**

---

## 1. 절대 원칙

아래 원칙은 반드시 지켜라.

### 1-1. 수정은 국소 패치로 끝내지 마라

특정 task 또는 특정 화면에서 발견된 오류를 단순히 그 산출물만 수정하는 것으로 끝내지 마라.  
반드시 아래 관점에서 재검토하여 시스템 차원의 반영 여부를 판단하라.

- 정책 반영 필요 여부
- 템플릿 반영 필요 여부
- machine spec 규칙 반영 필요 여부
- 테스트/회귀검증 반영 필요 여부
- 문서/절차 정합성 보정 필요 여부

### 1-2. 중간 산출물은 작업 중에만 사용하라

작업 중 생성되는 임시 분석물, 초안, diff 결과, 중간 리포트, 폐기된 시도본 등은 시스템의 장기 입력으로 남겨두지 마라.  
중간 산출물은 필요 시 생성하되, 작업 종료 후 반드시 정리하라.

### 1-3. archive는 절대 실행 입력이 아니다

archive 또는 로그성 폴더에 위치한 파일은 다음 analyze/build/fix/test 단계의 입력으로 사용되면 안 된다.  
단순 보관/추적용으로만 유지하라.

### 1-4. active만 공식 입력으로 인정하라

현재 시점에서 실제 실행에 사용되는 파일은 명확히 active 영역에만 존재해야 하며,  
working/archive 파일이 혼입되지 않도록 경로/정책/절차를 정비하라.

### 1-5. 문서와 실제 동작이 다르면 반드시 통일하라

README, ARCHITECTURE, commands, skills, policies, schemas, templates, tests 중 하나라도 서로 다른 규칙을 말하면 안 된다.  
발견 시 반드시 정리하여 하나의 운영 계약으로 맞춰라.

---

## 2. 이번 수정의 핵심 요구사항

이번 수정에서는 특히 아래 사항을 중심으로 시스템을 개선하라.

### 2-1. 정책 반영 절차 강화

현재 시스템에는 policy 관련 절차가 이미 존재한다.  
따라서 이를 제거하거나 무시하지 말고, 아래 방향으로 보완하라.

- 수정사항이 단건 처리로 끝나지 않고 정책으로 승격될 수 있도록 구조화
- 정책 반영 대상과 단순 산출물 수정 대상을 구분
- policy pending / active / archive 상태를 분리할 수 있도록 고려
- build 전 정책 동기화 또는 정책 참조 범위를 명확히 관리
- 정책이 문서, 템플릿, 테스트와 분리되어 따로 놀지 않도록 정합성 확보

### 2-2. 산출물 상태 분리

산출물은 최소 아래 3개 상태로 나누는 방향으로 정비하라.

- **active** : 현재 시스템이 실제 입력으로 사용하는 공식 산출물
- **working** : 작업 중 임시 생성물, 초안, 분석 중간본
- **archive** : 보관/추적용 로그성 산출물. 실행 입력 금지

필요하다면 task 단위, policy 단위, fix 단위로 적용하라.

### 2-3. cleanup / archive 절차 도입

각 단계 종료 후 working 산출물을 정리하는 절차를 설계하라.

최소한 아래 관점이 반영되어야 한다.

- 어떤 파일은 삭제 가능한가
- 어떤 파일은 archive로 이동해야 하는가
- 어떤 파일만 active로 승격되는가
- 다음 단계에서 참조 가능한 파일 범위는 어디까지인가
- archive를 시스템이 읽지 않도록 어떻게 보장할 것인가

### 2-4. 경로 및 참조 제한 강화

allowed path, manifest, command 절차, runtime 정책 등에서  
archive / history / log / old draft / discarded 자료가 실행 입력으로 섞이지 않도록 정리하라.

## 3. 수행 범위

다음 범위를 시스템 전체 관점에서 점검하고 수정하라.

### 3-1. 명령어/절차

필요 시 아래 명령어 또는 그 내부 절차를 수정하라.

- analyze
- build
- test
- fix
- 그 외 cleanup / sync / archive 관련 내부 절차

### 3-2. 정책

아래 계열의 정책 파일을 점검/수정하라.

- runtime 정책
- analyze/build/fix/test 관련 정책
- allowed path 관련 정책
- manual handoff 또는 예외 전환 정책
- policy patch 적용 절차

### 3-3. 스키마 및 계약

아래를 점검하여 실제 운영 절차와 맞춰라.

- manifest schema
- machine spec 관련 schema/rule
- 단계명(enum), 상태값, severity 정의
- active/working/archive 상태 정의 가능 여부

### 3-4. 폴더 구조

현재 폴더 구조를 분석하고, 필요 시 아래 방향으로 재구성하라.

- 실행 입력용 active 영역 명확화
- 중간 생성물용 working 영역 분리
- 로그/보관용 archive 영역 분리
- archive가 active 탐색 범위와 겹치지 않도록 조정

### 3-5. 문서

아래 문서가 실제 동작과 일치하도록 수정하라.

- README
- ARCHITECTURE
- command 문서
- skill 문서
- 정책 설명 문서
- 운영 규칙 문서

### 3-6. 테스트

테스트가 working/archive를 잘못 참조하지 않는지 점검하고,  
필요 시 아래를 반영하라.

- active 산출물만 검증 대상으로 사용
- archive는 검증 대상이 아님
- cleanup 이후에도 필요한 회귀 테스트는 유지
- 정책 변경이 회귀 테스트로 이어질 수 있도록 구조화

---

## 4. 산출물 구조 요구사항

작업 결과는 반드시 **“현재 시스템에 실제 반영되는 것”**과 **“기록용/중간용”**을 구분해서 만들어라.

### 4-1. 반드시 남겨야 하는 최종 산출물

최소 아래 성격의 최종 산출물이 있어야 한다.

1. **수정 결과 요약 문서**
   - 무엇을 왜 수정했는지
   - 어떤 문제가 있었는지
   - 어떤 범위까지 반영했는지

2. **구조/절차 변경 문서**
   - active / working / archive 정의
   - cleanup/archive 절차
   - 정책 반영 절차 강화 내용
   - 참조 가능/불가 경로 규칙

3. **실제 반영 파일**
   - 정책 수정
   - 명령어 수정
   - 스키마 수정
   - 템플릿/절차 수정
   - 테스트/검증 로직 수정

### 4-2. 중간 산출물 처리 원칙

중간 분석물은 자유롭게 생성 가능하나, 최종적으로는 아래 중 하나로 처리하라.

- active로 승격
- archive로 이동
- 삭제

working에 불필요한 파일이 계속 남지 않도록 하라.

### 4-3. archive 처리 원칙

archive는 아래 조건을 만족해야 한다.

- 과거 추적 가능
- 실행 입력 금지
- 기본 탐색 대상 제외
- 사람이 필요할 때만 열람 가능

---

## 5. 설계 방향

수정 시 아래 설계 방향을 따르라.

### 5-1. 상태 기반 산출물 관리

파일을 단순히 생성/수정하지 말고,  
“이 파일이 현재 active인지, working인지, archive인지”가 명확히 드러나는 구조로 정리하라.

### 5-2. 시스템 입력 최소화

에이전트가 다음 작업에서 읽는 파일 수를 최소화하라.  
최신 공식 입력만 참조하도록 하라.

### 5-3. 문맥 오염 방지

과거 초안, 폐기본, 중간 분석물, 실패 시도본, 오래된 비교 리포트가  
새 작업의 판단 근거로 섞이지 않도록 차단하라.

### 5-4. 수정의 승격 구조

수정사항은 가능하면 아래 수준 중 어디에 반영되어야 하는지 판단하라.

- Local patch
- Type-level rule
- Global policy

판단 근거 없이 국소 수정만 하지 마라.

### 5-5. 재발 방지 우선

이번 수정으로 끝나지 않고 이후 유사한 상황에서도 같은 문제가 줄어들도록 설계하라.

---

## 6. 기대 작업 방식

아래 방식으로 작업하라.

### 6-1. 먼저 현재 구조를 분석하라

아래를 우선 파악하라.

- 현재 policy 반영 절차가 어디까지 존재하는지
- working 성격의 산출물이 어디서 생기는지
- archive 또는 로그 분리 구조가 이미 있는지
- 명령어별 입력 참조 범위가 어떻게 되는지
- 어떤 문서/정책/스키마가 서로 불일치하는지

### 6-2. 그 다음 개선 설계를 제안하라

분석 결과를 바탕으로 아래를 설계하라.

- 개선된 폴더 구조
- 파일 상태 분류 기준
- cleanup/archive 절차
- command별 입력/출력 정책
- policy active/pending/archive 운영 방식
- allowed path 또는 manifest 반영 방안

### 6-3. 이후 실제 수정하라

설계만 하지 말고, 실제로 반영 가능한 파일을 수정하라.

### 6-4. 마지막에 결과를 정리하라

최종적으로 아래를 명확히 남겨라.

- 수정된 파일 목록
- 변경 이유
- 운영 방식의 변화
- 이후 사용자가 따라야 할 규칙
- 주의사항 및 남은 과제

## 7. 출력 형식

최종 답변 및 산출물은 아래 형식을 따르라.

### 7-1. 필수 포함 항목

반드시 아래 항목을 포함하라.

#### A. 현행 문제점 분석

- 정책 절차의 현재 상태
- 문맥 오염 가능 지점
- 중간 산출물 누적 문제
- 참조 범위 혼선
- 정합성 불일치 항목

#### B. 개선 설계안

- active / working / archive 정의
- 폴더 구조 개편안
- cleanup/archive 절차
- 정책 반영 절차 강화안
- 명령어별 입력 참조 정책
- archive 차단 방식

#### C. 실제 수정 내용

- 수정 파일 목록
- 파일별 수정 요약
- 정책/문서/명령어/스키마/테스트 반영 내용

#### D. 운영 규칙

- 앞으로 어떤 파일이 공식 입력인지
- 중간 산출물은 어디에 두는지
- 작업 종료 후 무엇을 정리해야 하는지
- archive는 언제 열람하는지

#### E. 잔여 리스크

- 아직 미완료된 부분
- 추가 개선이 필요한 부분
- 향후 고도화 방향

### 7-2. 형식 제약

- 답변은 보고서형 마크다운으로 작성하라.
- 개조식으로 명확히 정리하라.
- 추상적 설명만 하지 말고 실제 반영 가능한 수준으로 작성하라.
- 필요 시 디렉토리 트리, 규칙표, 절차표를 포함하라.
- “검토 필요”, “보류”, “권장”, “필수”를 구분해서 표시하라.

## 8. 금지 사항

아래는 금지한다.

- 단순히 “archive 폴더를 만들자” 수준의 피상적 제안만 하는 것
- 중간 산출물을 계속 active 영역에 남겨두는 것
- archive를 시스템 입력에서 완전히 배제하지 않는 것
- 정책 절차가 이미 있는데 무시하고 새 체계만 덧붙이는 것
- README/ARCHITECTURE/실제 동작이 계속 불일치한 상태로 두는 것
- 이번 수정사항을 특정 task 전용 패치로만 끝내는 것
- 불필요하게 과도한 중간산출물을 추가 생성하는 것
- working과 archive의 차이를 모호하게 두는 것

---

## 9. 최종 목표

이번 작업의 최종 목표는 다음과 같다.

> **수정사항이 시스템 전체에 구조적으로 반영되고, 중간 산출물이 문맥을 오염시키지 않으며, archive는 로그로만 남고, 실제 실행은 active 공식 산출물만 참조하는 지속가능한 시스템으로 정비하는 것**

이 목표를 기준으로 분석하고, 설계하고, 실제 수정하라.

---

필요 시 아래 문장을 작업 시작 시 내부 판단 기준으로 사용하라.

> “이번 수정은 개별 오류 수정이 아니라, 정책 반영 절차를 유지·강화하면서 시스템 산출물의 생명주기를 통제하고 문맥 오염을 막는 구조 개선 작업이다.”
